Infrastructure

We use Unity and CMock for unit testing. They really are very simple frameworks which just add a few macros and functions for assertions and evaluating test cases.

Build Process

The developer runs 

make test

in the terminal. In the make test target it travels to the directory common-all/tests/. Then it runs the Makefile in that directory which subsequently calls the rakefile in common-all/tests/.

The rakefile then creates the object files for each C file involved these include:

  • Test Files, files like test_controlStateMachine.c, which hold all of our test cases
  • Mock/Fake files, the files we make to emulate the HAL and FreeRTOS functions which CMock and Unity cannot parse nor compile as it is target specific.
  • Tested file, file of which we are testing for example controlStateMachine.c. These are the files with the functions we are testing.

The rakefile links these all together as well as the Unity/CMock runner file to create a test executable. The rakefile then runs all of the test executables to check if all of the tests passed.

Setting up New Repository for Testing

  1. Make sure all of the git submodules have been setup and initialized
    1. Run git submodule update --init --recursive
  2. Ensure that you have all the necessary dependencies
    1. You need:
      1. Ruby
      2. Rake
        1. gem install rake
  3. Create the following folders in the root directory of the repository:
    1. tests/
    2. tests/build/
  4. In tests/build/ create a file called .gitkeep (this is an empty file and is just so the tests/build/ directory gets pushed to Bitbucket).

  5. In the .gitignore file (found at the repository's root directory), add these 2 lines:

    tests/build/*
!tests/build/.gitkeep

    This ensures that any test output generated in tests/build/ doesn't get pushed to Bitbucket, but the tests/build/ directory still does.

  6. In tests/ create the config file test.yml
    1. This should contain compiler flags, directories of source files, location of HAL files, etc.
    2. Ensure that all paths in the test.yml file exist
      1. For repositories with the directory Cube-F7-Src-respin, use it over Cube-F7-Src and Cube-Nucleo_Src
    3. See Test.yml example below
  7. In tests/ add your test files
    1. See below documentation for more info on writing these unit tests.

Current Task for Unit Tests

Currently, there are unit tests (found here) for the common-all repository which is a repository with common functions used between all boards on the car (PDU, VCU, DCU, BMU, etc.). This means that the drivers for the finite state machines are being validated but we need to make sure that the transitions work for each board.

Example for the VCU: When we receive a CAN message from the DCU that says "HV_Enable", does the board transition to the HV_Enable state? Does the associated functionality change?


To familiarize yourself with state machines and how they work refer to the page here


Some functions that operate on the state machines are:

  • To process an event in the state machine queue use fake_mock_fsm_run(). This will create a separate thread to process the transition.

    pthread_t fsm_id = fake_mock_fsm_run(&motorFsmHandle);
end_task(fsm_id);

  • To check the state of the state machine using its handle call fsmGetState()

    uint32_t state = fsmGetState(&motorFsmHandle);


What you need to do is:

  1. Find the file in the repository you were assigned called "controlStateMachine.c" which is under the Src/ directory of the board repository

  2. Checkout the "develop" branch for the board-specific repo and the "WFE-807-CMock_Infra" branch in common-all 

    cd 2018_XXX //Where xxx stands for pdu,dcu,bmu,vcu
git checkout develop
cd common-all
git checkout WFE-807-CMock_Infra
cd test-core
rm -rf CMock
git clone --recursive https://github.com/ThrowTheSwitch/CMock.git
cd CMock
bundle install #Make sure the bundle application is installed on your system
  3. Read through each function in the file one at a time and identify functions that call for a change in a state machine. (If you can not find the function in the repo you are in, don't forget to check the common-all repo)

  4. Create a file called test_controlStateMachine.c in the directory tests/ (ex. 2018_pdu/tests/test_controlStateMachine.c) of the board repo (you may have to create the file) and make sure it contains the starter code below.

    #include "unity.h"

#include "controlStateMachine.h"

#include "Mock_stm32f7xx_hal.h"
#include "Mock_userCan.h"
#include "Mock_DCU_can.h"

#include "Mock_debug.h"
#include "fake_debug.h"

#include "Mock_queue.h"
#include "fake_queue.h"

#include "Mock_main.h"
#include "fake_main.h"

#include "state_machine.h"

#include "fake_state_machine_interface.h"

#include "Mock_canReceive.h"

#include "Mock_can.h"
#include "fake_can.h"

#include "fake_timer.h"
#include "fake_tasks.h"

#include "Mock_gpio.h"
#include "fake_gpio.h"

#include "Mock_watchdog.h"

// These CAN variables have to be initialized for use in our tests
volatile int64_t ButtonEMEnabled;
volatile int64_t ButtonHVEnabled;

void setUp(void)
{
	fake_mock_init_queues();
	_Error_Handler_Stub(fake_Error_Handler);
	fake_mock_init_debug();

}

void tearDown(void)
{
}
  5. You will need to identify all of the CAN variables and define them with the volatile keyword. In the sample code above these are defined above the setup() function.
  6. Copy the name of the original function from step 2 but prepend the string "test_" to it. For example, if the function name is "fsmInit()" call the new function "test_fsmInit()". The return type of these functions should be of type void
  7. For every condition in this function that needs to be verified, write a condition for it and wrap it with TEST_ASSERT_TRUE. This means if statements and variable values should have a TEST_ASSERT statement.
    1. Example: TEST_ASSERT_TRUE(getEMState() == EM_State);
  8. Once you are done going through a function in controlStateMachine.c, try testing your code by running "make test" in the terminal. You should see your test show up in green indicating that it passed
  9. Repeat steps 4-6 for every function in controlStateMachine.c


Example for the PDU:

  1. Open the file 2018_pdu/Src/controlStateMachine.c
  2. The first function of interest is motorControlInit()
    1. In this function, you will notice that the function fsmInit() is called so look for the implementation of the function to understand what it does.
    2. The function can be found in the common-all repo here. Reading through the function you will realize that a queue is created and variables such as startingState are verified and then set.
    3. After understanding the code, we will realize that the state machine kept in the variable motorFsmHandle should be initialized to the state MTR_STATE_Motors_Off. We will use this information to write our unit test.

  3. - 6. The function added to the file test_controlStateMachine.c is shown below:

    void test_motorControlInit()
{
    TEST_ASSERT_TRUE(motorControlInit() == HAL_OK);
	TEST_ASSERT_TRUE(fsmGetState(&motorFsmHandle) == MTR_STATE_Motors_Off);
}
