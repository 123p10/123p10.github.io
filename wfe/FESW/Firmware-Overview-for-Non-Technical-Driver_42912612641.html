Firmware : Firmware Overview for Non-Technical Driver

Created by Owen Brake, last modified on Feb 05, 2022

Good reference for Acronyms, Terms: Overview of Acronyms, Terms, Definitions

Overview of Car Architecture

CAN - Control Area Network

  • CAN Messages

    • Well-defined messages broadcasted on a single CAN bus

    • Message definition stored in “DBC“ files which are parsed and “stored“ on each board

    • Usually used to indicate an event or regularly broadcast a signal, e.g:

      • Report battery voltage every X ms

      • Indicate that the user has pressed a button

  • DTC (Diagnostic Trouble Code) Messages

    • Special type of CAN message

    • Indicates something has gone wrong and informs all the boards, so they can take corrective action

    • Broken into severities (from low to high):

      • Warning

      • Error

      • Critical

      • Fatal

    • The most up-to-date list of DTCs in use are detailed here: https://bitbucket.org/waterloohybrid/firmware/src/main/common/Data/DTC.csv

      • More work needs to be done to clean up DTC names

  • Heartbeats

    • Each board sends a message over CAN every 1s called a “heartbeat“

    • Each board will check that it received a heartbeat message from every board. If 2.5s have passed and a board has not received a heartbeat from another board it will send a Fatal DTC.

      • This will shut the car down

    • Possible causes of missed heartbeat:

      • Electrical failure causing damage to transmitter/receiver board

      • Firmware crashed/hanging

      • CAN bus damage

Boards

BMU - Battery Management Unit

  • 1 BMU board, interfaces with 6 AMS boards which:

    • Balance battery cells

    • Measure Cell Voltage/Temperature

  • BMU monitors and controls the battery

PDU - Power Distribution Unit

  • Manages distribution of low voltage power to all the boards

  • Selects which Low Voltage (LV) power source to use:

    • Lithium Polymer LV battery

    • DC-DC, to step down voltage from the battery

VCU - Vehicle Control Unit

  • Central processing unit of entire car

  • Controls:

    • Beaglebone

    • Motor Controllers

    • Pedals (Throttle/Brake)

DCU - Dashboard Control Unit

  • Controls vehicle dashboard

  • User buttons

WSB - Wheel Sensor Board

  • Least important board

    • Implementation very weak, as of right now (W2022)

  • Reads sensors connected to each wheel and reports data to the other boards over CAN

    • Hall Effect Sensor (Wheel speed)

    • Tire Temperature

    • Strain Gauge

Beaglebone

  • Mounted on VCU

  • Performs analytics logging

    • CAN Message

    • GPU

    • IMU (Inertial Measurement Unit)

  • Dashboard Heads up Display

    • Dashboard is still WIP, planning to be working by competition

    • Will display DTC messages when they appear

    • Also expecting to have vehicle speed and battery SoC (state of charge in a percentage)

Car Operation

Getting Car to Drive

  • Much of this is retrieved from here: Startup Procedure , I’ve adapted it to be more appropriate/simpler for drive use cases

Step-By-Step

1. Switch on the GLVMS (Grounded Low Voltage Master Switch). This means switching the red switch to the left.

2. Ensure that the red emergency stop buttons are not pressed in.

3. Press the button labelled "HV Enable" (High Voltage), it should be the top button on the dashboard. It should light up red when pressed

4. The TSAL (Tractive Systems Active Light) should now be flashing red to indicate that the system is in High Voltage

5. Before starting the EM (Electric Motors) a few conditions must be reached.

1. HV must already be on

2. The brake pedal must be pressed

3. The throttle pedal cannot be pressed

4. There is sufficient brake pressure

6. Once these conditions are reached. Press the EM Enable Button this is the lower right button on the dashboard as shown. The button should light up blue

7. The car is now ready to drive. Release the brake pedal and you can now gently apply pressure to the throttle to drive the car.

Endurance Mode

  • This is a WIP feature that is subject to change

  • The details are quite technical and can be viewed here: Endurance Mode

  • A feedback loop adjusts the power delivered to the car every lap

    • Uses PI loop

      • If one drives a lap significantly faster than expected the feedback loop will make a significant correction

      • If one drives slightly faster for many laps, the feedback loop will slowly make corrections

  • If the car goes a lap too fast, the feedback loop will try and limit the power to slow down the car so it is ensured the car will complete the 44km course

  • The feature can be toggled by one of the buttons on the left side of the dashboard (TBD)

  • The Battery SoC (State of charge as a percentage) will be reported on the dashboard heads up display

  • Ideally, the driver can just drive as normal and be guaranteed to complete the course.

Car Diagnostics

Flashing the Car

  • Setup your system to have openOCD, arm-none-eabi-gcc, Python…

    • Setup your system following these instructions: Tool Setup

      • Probably unnecessary for non-firmware members

  • Clone the firmware repo: https://bitbucket.org/waterloohybrid/firmware/src/main/

  • Plug in the Sophia board with the 12 pin hirose connector as shown below into the computer and the board to be flashed

  • Run 

    make load LOAD_TARGET=<board_name, bmu, pdu, vcu>

Stopping the Car

While as a Driver

  • Press the Driver E-Stop (a.k.a Cockpit BRB (Big Red Button))

    • This shuts off the high voltage in the system, so any driving will immediately cease.

    • It should be able to be reset by the driver, so driving can resume when the button is pulled out

  • Press EM Enable, then HV Enable again to turn off the EM and HV

    • This is the “standard“ shutdown method

    • Will not shut down LV

    • “Safest“ method to shutdown in terms of least likely to cause issues

While as a Bystander

  • Hit the Side Mounted BRBs (Big Red Button)

    • Shuts off HV in the system, prevents EM from driving

Debugging, Detecting Failures

DTCs on Heads up Display

The heads up display looks like below:

  • Current DTC messages aren’t great for an end user. We will work on updating them to be properly readable and understandable

    • Note currently we display the most recent DTC messages, not filtered for priority. The intention is that DTCs are fairly rare but it’s best the driver sees the most recent ones so they can make corrective action to new information.

      • Also high priority DTCs cause system failure so not much point in corrective actions

  • DTC messages color coded.

    • FATAL is red and it will cause major system shutdowns.

      • Ex. PDU is turning off all boards or one of the boards has not responded to a heartbeat

    • CRITICAL is really bad, will most likely cause significant performance degradation but car may still technically function

      • Ex. Brake Pressure or Throttle sensors are broken, so will stop VCU from functioning

    • ERROR is orange, there are no ERROR DTCs right now. But they would (probably) represent correctable conditions, though conditions which may degrade performance

    • WARNING is yellow, it’s a DTC but not something to freak out about. Often can be ignored, but represents behavior to monitor. Don’t expect any performance changes.

      • Ex. Battery voltage is a little too low/high, but not critically out of range.

CAN Logging

  • The beaglebone will(is expected to) log all CAN messages as soon as it’s turned on

    • Should turn on as soon as VCU turns on

  • These CAN logs are stored on the beaglebone as simple text files on the beaglebone file system

    • Should be at /home/wfe/logs/<timestamp>.txt

  • The beaglebone can be remotely SSHed into to get CAN logs

    • Simply power on the GLV for the car (turn the GLVMS)

    • SSH and then the beaglebone can be SSHed into (Guide here: Connecting to and Running code on the Beaglebone ). Note the guide may need to be updated for new changes.
