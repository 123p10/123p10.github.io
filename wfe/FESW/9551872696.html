Firmware : Firmware Workflow (using Git)

Created by Former user, last modified by Former user on Oct 28, 2020

Good version control forms the core of the firmware workflow. In order to use version control in an effective manner, we use the gitflow git branch strategy. See this page for more info: https://www.atlassian.com/git/tutorials/comparing-workflows/gitflow-workflow (If you need an intro to git, see the Git Tutorial page).

The parts of the various parts of the workflow are describe in more detail below, and the steps in more detail on the linked page above. However, a short summary is:

  1. Create a branch for the feature/bug/etc you are working on
  2. Work on the feature, making commits as you get working code
  3. Once the feature is complete, perform tests to verify it works as desired
  4. Create a page on confluence documenting the feature
  5. Get your code reviewed by other through pull requests
    1. If necessary, make changes to your code to get it accepted, and update the confluence page(s) as appropriate
  6. Once your pull request is accepted, merge the pull request to the appropriate branch

Branches

The branches should follow the gitflow branch strategy (https://www.atlassian.com/git/tutorials/comparing-workflows/gitflow-workflow)

All branches should use the branch name scheme as follows: <branch type>/<descriptive branch name>. Branch type is one of the git flow branch types, and then it is up to you to choose a descriptive branch name.

The branch types are (see the gitflow link for more info on the types):

  • bugfix/
  • feature/
  • hotfix/
  • release/

The main branch types you'll likely use are bugfix and feature branches. They differ in what they are used for, the branches that they can be created off and merged into, and the branch name prefix

Feature Branches

These branches are used to work on implementing new features to the codebase (as opposed to bugfixes, where you are making changes to an existing feature to fix an issue with it). They should only be branched off of the develop branch, and only merged back into the develop branch. As well, their branch name should be prefixed with feature/, for example the name could be feature/brake-light for working on implementing the brake light.

Bugfix Branches

These branches are used to work on fixing bugs that affects existing code. They can be branched of either the develop branch, or a release branch, and back into either of those (normally the same branch that they are branched off of). As well, their branch name should be prefixed with bugfix/, for example the name could be bugfix/fix-brake-light for fixing a bug with the existing brake light code.

Commit Messages

All commit messages need to reference an issue that they are related too. Since a branch should be created for each new feature or bugfix, a commit should only ever be related to one issue.

Pull Requests

Once you have finished you work on a bug/feature/etc in you branch, you will want to merge it into the develop branch. To do this, use a pull request on bitbucket. Navigate to the branches section of the Bitbucket repository, and select create pull request. Alternatively, when you push the branch to Bitbucket you will get a message with a link to create the pull request.

In filling out the pull request, try to fill in as much of the default template as possible. The mandatory fields are the title, summary, subject matter experts (should be at least you hopefully...), the description, and the testing done. If there are no relevant rules or confluence pages (maybe you should create one though?), its ok to put N/A. For reviewers, there should be a default reviewer added for the repository, optionally you can add more reviewers who you think should review your change. Finally, select close branch on merge.

The next step is to make sure you get approval for the pull request before you merge it. You need to get approval from at least one default reviewer before you merge the pull request. Once you have that, perform the merge (make sure Bitbucket doesn't give you a warning that there is merge conflict, if there are you need to fix those first).

Once the pull request is merged, you need to link the pull request in the relevant issue, see https://confluence.atlassian.com/bitbucket/mark-up-comments-305037452.html for how to do that, but basically just make a comment with test "pull request # <pull request number>"
